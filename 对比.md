## Veritas vs ELIS 对比分析

### ELIS 项目概览

ELIS (Scientific Integrity System) 是面向科学图像诚信的开源系统，致敬 Dr. Elisabeth Bik。定位于科研图像取证的通用平台，采用 **FastAPI + Celery + MongoDB + Redis** 的经典微服务架构，前端 Vite/React/Tailwind。所有视觉取证模块（panel-extractor、copy-move、TruFor、CBIR、provenance-analysis、watermark-removal）均以独立 Docker 容器运行并通过 Celery Worker 调度。ELIS 是多租户系统，提供完整的 JWT 鉴权、用户管理、文档/图像 CRUD、存储配额、annotation 和 admin 功能。使用 **AGPL-3.0** 许可证开源。

### Veritas 项目概览

Veritas 是内部实验室论文风控工具，聚焦干实验论文投稿前技术复核。采用 **CLI-first** 设计，核心是 `audit-paper` 闭环流水线。后端使用 stdlib HTTP + SQLite/PostgreSQL (pgvector)，前端 Vite/React/Tailwind 内测工作台。核心差异化：**(1) Source Data 全系列检测**、**(2) Agent 编排**（opencode + Tool Registry）、**(3) PaperFraud rule match**、**(4) 结构化 HTML 报告**。通过 ELIS adapter 复用 YOLOv5/RootSIFT/SILA/TruFor，自研 overlap_reuse。不开源（内部工具）。

## 关键发现

**发现 1：Veritas 在 Source Data 和 Agent 编排方面确实超越 ELIS，但在 CBIR/Milvus 和完整 TruFor 方面仍有缺口**

"Veritas 是 ELIS 的超集"这个命题**部分成立**。Veritas 在 Source Data 检测、Agent 编排、结构化报告、PaperFraud rule match、overlap_reuse 等维度确实是 ELIS 没有的新能力。但在视觉取证方面，Veritas 的 **CBIR/Milvus 向量检索**（跨论文相似图搜索）和 **TruFor 有效执行**（需 GPU）仍是缺口。溯源图方面，Veritas 的 pairwise provenance_graph 不等同于 ELIS 的递归 BFS provenance-analysis。

**发现 2：两个项目的定位根本不同**

ELIS 的核心是"让科研诚信工具民主化"，面向全球用户，多租户 SaaS，用户手动触发各分析步骤。Veritas 的核心是"帮导师在投稿前发现学生数据问题"，CLI-first，自动编排审计流水线。Veritas 不是 ELIS 的"升级版"，而是在不同定位上吸收了 ELIS 的视觉取证能力。

**发现 3：Veritas 技术架构更轻量，ELIS 多租户和用户管理更完整**

ELIS 有完整的 JWT 鉴权、用户 CRUD、存储配额、annotation 系统、admin dashboard。Veritas 的 Web 鉴权只有 NoAuth/BearerToken/BasicAuth，没有完整的用户管理界面。这是内部工具和开源 SaaS 的定位差异。

**发现 4：ELIS 前端更成熟，Veritas 前端有独特业务页面**

ELIS 前端 128 源文件，含 CBIR search、copy-move、provenance、annotation、admin。Veritas 前端 26 源文件，含 VisualForensicsPage、OverlapGraph、InvestigationBoardPage 等独特业务页面。技术栈相同（Vite/React/Tailwind/d3），业务逻辑不同。

**发现 5：Veritas 的 ELIS adapter 复用策略是正确的**

Veritas 通过 subprocess/Docker wrapper 调用 ELIS 的 system_modules，而非接入 ELIS FastAPI/Celery/MongoDB/Redis 主服务。这避免了重基础设施依赖，保持了轻量架构，同时复用了 ELIS 的算法能力。这是一个合理的工程决策。

---

## 结论

**"Veritas 是 ELIS 的超集"这个说法需要修正：**

- **在 Source Data 检测、Agent 编排、结构化报告方面**：Veritas 超越 ELIS，这是 ELIS 没有的能力
- **在视觉取证核心能力方面**：Veritas 通过 adapter 复用了 ELIS 的大部分能力（panel extraction、copy-move、TruFor schema），但 **CBIR/Milvus 跨论文检索**是最大缺口
- **在溯源图方面**：Veritas 是 pairwise matching，ELIS 是递归 BFS，能力不对等
- **在定位方面**：两个项目定位不同，不是简单的超集关系

Veritas 的真正差异化是 **Source Data 检测 + Agent 编排 + 垂直领域聚焦**，而不是"ELIS 的所有功能 + 更多"。

好，SSCD + pgvector 已有基础实现。下面是我的评估：

## 详细分析

### 计划 1：stdlib HTTP → FastAPI

**评估：不值得做**

| 维度          | 分析                                                                                                                |
| ------------- | ------------------------------------------------------------------------------------------------------------------- |
| **当前状态**  | stdlib HTTP 运行稳定，Web P1 内测可用                                                                               |
| **迁移成本**  | 高。需要重写所有路由、依赖注入、中间件、测试；涉及 20+ 文件                                                         |
| **收益**      | 低。FastAPI 的优势（自动 OpenAPI、异步、类型校验）对内部工具价值有限                                                |
| **风险**      | 破坏现有 Web 功能，测试需要重写                                                                                     |
| **ELIS 对比** | ELIS 用 FastAPI 是因为它是多租户 SaaS，需要 OpenAPI 文档和异步高并发。Veritas 是内部工具，用户量小，stdio HTTP 足够 |

**建议**：除非有明确的性能瓶颈或功能需求（如 WebSocket、流式响应），否则不迁移。把精力放在核心功能上。

---

### 计划 2：用户管理 + annotation + admin

**评估：过度设计**

| 维度                | 分析                                                                                                                |
| ------------------- | ------------------------------------------------------------------------------------------------------------------- |
| **当前状态**        | Veritas 有 NoAuth/BearerToken/BasicAuth 三种模式，满足内部使用                                                      |
| **需求真实性**      | Veritas 是**内部工具**，用户是实验室导师和学生，不需要多租户、存储配额、admin dashboard                             |
| **annotation 系统** | ELIS 的 annotation 是面向全球开源社区的协作标注。Veritas 的场景是"导师审查学生数据"，不需要复杂的 annotation 工作流 |
| **开发成本**        | 高。需要数据库 schema、API、前端页面、权限系统、测试                                                                |
| **ELIS 对比**       | ELIS 做这些是因为它是 SaaS 产品。Veritas 的核心价值是 Source Data 检测 + Agent 编排，不是用户管理                   |

**建议**：不做。Veritas 不是 SaaS，不需要多租户和用户管理。如果未来需要权限控制，可以基于现有的 BasicAuth 扩展，不需要照搬 ELIS 的完整系统。

---

### 计划 3：SSCD + pgvector 完善

**评估：优先做**

| 维度             | 分析                                                                                                                                                                                    |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **当前状态**     | SSCD encoder 已实现（`engine/embeddings/sscd.py`），pgvector 已在数据库模型中（`web/backend/veritas_web/models.py`）                                                                    |
| **缺口**         | CBIR 搜索端未实现；SSCD 未接入主链路；前端无 CBIR search page                                                                                                                           |
| **与 ELIS 对比** | ELIS 用 SSCD + Milvus，Veritas 用 SSCD + pgvector。pgvector 是 PostgreSQL 扩展，不需要额外的 Milvus 微服务，架构更轻量                                                                  |
| **实现路径**     | 1. 完善 embedding 索引管线（panel → SSCD → pgvector）<br>2. 实现 CBIR 搜索 API（相似度查询）<br>3. 前端增加 CBIR search page<br>4. 集成到 Agent investigation（agent 可选择 CBIR 工具） |
| **收益**         | 补齐跨论文相似图检索能力，这是 ELIS 有而 Veritas 没有的核心视觉取证能力                                                                                                                 |

**建议**：P1 优先级。已有 60% 基础，补齐 40% 就能达到 ELIS 的 CBIR 能力水平，且架构更轻量（不需要 Milvus）。

---

### 计划 4：溯源图升级为递归 BFS + MST

**评估：优先做**

| 维度          | 分析                                                                                                                                                                                                   |
| ------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **当前状态**  | Veritas 的 `visual.provenance_graph` 是 pairwise matching（dHash pre-filter + RootSIFT pairwise）                                                                                                      |
| **ELIS 方案** | G2NN + MAGSAC++ 递归 BFS + MST（最小生成树）；可以递归扩展溯源链                                                                                                                                       |
| **能力差异**  | Veritas 只能检测两两关系，ELIS 可以递归发现"图 A 和 B 相似，B 和 C 相似，C 和 D 相似"的链条                                                                                                            |
| **实现路径**  | 1. 复用 ELIS 的 provenance-analysis 算法（G2NN + MAGSAC++ + BFS + MST）<br>2. 通过 adapter 方式集成（subprocess 调用 ELIS 的 provenance-analysis 容器）<br>3. 前端增加递归溯源图可视化（vis.js 或 d3） |
| **收益**      | 视觉取证的核心差异化能力；可以发现更复杂的图像复用模式                                                                                                                                                 |

**建议**：P1 优先级。这是 ELIS 的强项，Veritas 应该通过 adapter 复用 ELIS 的 provenance-analysis 算法，而不是自己重新实现。

## 总结

Veritas 的核心价值是 **Source Data 检测 + Agent 编排 + 垂直领域聚焦**，不是"照搬 ELIS 的所有功能"。下一步应该聚焦在：

1. **补齐 CBIR 能力**（SSCD + pgvector 完善）— 这是 ELIS 有而 Veritas 没有的核心能力
2. **升级溯源图**（复用 ELIS provenance-analysis）— 这是视觉取证的核心差异化

不要做：
- FastAPI 迁移（成本高，收益低）
- 用户管理/annotation/admin（过度设计，不符合定位）
